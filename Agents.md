# 프로젝트 협업 규칙

이 문서는 이 프로젝트에서 사용자와 AI 에이전트가 함께 작업할 때 따를 규칙을 기록합니다.

## 개인 작업 규칙

- 코드 최신화를 요청받으면 현재 브랜치와 커밋, 작업 트리 상태를 확인하고, 이전 동기화 이후의 PR 및 커밋 내역을 기능별로 요약한다.
- 사용자가 명시적으로 요청하기 전에는 커밋, 푸시, 배포를 진행하지 않는다.
- 사용자에게는 한국어 존댓말로 간결하고 직접적으로 답변한다.

## Git 및 배포

- GitHub CLI 인증 계정은 `NohMyungWook`이며, PR 제목·본문·머지 내역을 조회해 요약에 활용한다.
- 코드 최신화 요청은 항상 `main` 브랜치로 전환한 뒤 `origin/main`을 pull 받으라는 의미로 해석한다. 이후 현재 브랜치·커밋·작업 트리 상태와 변경 내역을 요약한다.
- 단순 코드 최신화 요청에는 브랜치 전환, `pull`, 상태·변경 내역 확인만 수행한다. 코드를 변경하지 않았다면 빌드와 lint를 실행하지 않는다.
- `main` 브랜치에서 특정 작업을 요청받으면, 작업 시작 전에 해당 작업 목적에 맞는 새 작업 브랜치를 생성하고 그 브랜치에서만 변경한다.
- 커밋을 요청받으면 기능 단위로 나누고, 현재 작업 브랜치에만 커밋한다. `main` 직접 push 및 PR 자동 머지는 하지 않는다.
- PR 생성을 요청받으면 `main`을 base로 사용하고, 본문을 `## 요약`, `## 변경`, `## Test` 구조로 작성한다. 변경 목적과 사용자 영향, 주요 파일별 변경점, 실제 실행한 검증 명령을 구체적으로 기록한다.
- 실행하지 않은 검증은 PR 체크박스에서 완료로 표시하지 않는다. DB·배포·마이그레이션 관련 주의사항이 있으면 별도 항목으로 명시한다.
- 배포를 요청받으면 배포 완료 상태와 실제 운영 alias를 확인하고, 관련 API나 화면을 가능한 범위에서 운영 환경 기준으로 검증한다.
- 기존 작업 트리의 사용자 변경사항을 임의로 삭제하거나 덮어쓰지 않는다.

## 구현 및 UI 원칙

- 사용자가 요청하지 않은 기능, 문구, 색상, 동작을 임의로 변경하지 않는다. 리팩터링에서도 기존 동작과 디자인을 보존한다.
- UI 변경은 모바일, 일반 웹, 와이드 화면을 함께 점검한다. 내부 콘텐츠에는 반응형 여백을 두되 페이지 배경·내비게이션을 고정 폭으로 제한하지 않는다.
- 모바일에서 텍스트 잘림, 불필요한 2분할, 레이아웃 흔들림, 크기 변화가 생기지 않도록 한다.
- 전체 기본 글꼴은 Pretendard를 유지한다.
- 한 파일이 지나치게 커지면 페이지·컴포넌트·스타일·도메인 함수 기준으로 분리하되, 과도한 추상화는 피한다.
